## 1. 면접 답변 카드 — 30초 안에 말할 수 있게

TRD 3.19 의 12 질문 중 핵심 8개를 직접 답으로 채우세요. 한 답변은 **3~4 문장 / 30초** 분량.

### Q1. 이 프로젝트를 왜 만들었나요?

힌트: SI 업무에서 자주 등장하는 SCM/구매·판매 도메인을 이해하기 위해, 헤더-라인 + 다단계 상태 전이 + 권한 분리를 한 번에 다룬다.

> A:

### Q2. SCM 시스템에서 가장 중요한 도메인은?

힌트: 발주 헤더-라인 구조 + 상태 전이(DRAFT → REQUESTED → APPROVED → RECEIVED) + 거래처 유형 검증.

> A:

### Q3. User 와 Partner 를 왜 분리했나요?

힌트: User 는 로그인 계정(인증/권한), Partner 는 거래처 마스터(공급사/고객사). 라이프사이클과 권한 책임이 다르다.

> A:

### Q4. 거래처 유형(SUPPLIER/CUSTOMER/BOTH) 을 어떻게 관리했나요?

힌트: enum + EnumType.STRING, Service 의 `validatePartnerType` 으로 다층 방어.

> A:

### Q5. 발주 상태값은 어떻게 관리했나요?

힌트: enum + EnumType.STRING, 상태 머신 (DRAFT→REQUESTED→APPROVED/REJECTED→RECEIVED, CANCELED). 도메인 메서드 안에 상태 가드.

> A:

### Q6. 헤더-라인 구조에서 가장 중요한 검증은?

힌트: 라인 검증을 헤더 save 전에 끝낸다. 총금액 = 라인 합계 도메인 메서드. `cascade = ALL + orphanRemoval` 의 효과/위험.

> A:

### Q7. @Transactional 은 어디에 적용했고, 왜 적용했나요?

힌트: 발주 헤더+라인 동시 저장, 발주 승인 상태 변경 + approvedAt, 수주 출고 + shippedAt 갱신.

> A:

### Q8. Entity 를 직접 응답하지 않고 DTO 를 사용한 이유는?

힌트: 비밀번호 노출 방지, LAZY 직렬화(LazyInitializationException) 문제, API 계약 안정성.

> A:

### Q9. 관리자/일반/매니저 권한은 어떻게 구분했나요?

힌트: UserRole enum (USER/ADMIN/MANAGER), `@PreAuthorize("hasRole('...')")` 1차 + Service 가드 2차.

> A:

### Q10. 가장 어려웠던 문제와 해결 방법은?

힌트: TRD 3.16 의 트러블슈팅 1건을 골라, 4단계(문제-원인-해결-배운점)로 압축.

> A:

## 3. 커밋 메시지 컨벤션 (TRD 3.18)

### 타입 (직접 채우기)

| 타입 | 의미 | 예시 |
|---|---|---|
| feat | 새 기능 | `feat: 발주 헤더+라인 저장 트랜잭션 적용` |
| ____ | 버그 수정 | `fix: 이미 APPROVED 인 발주 재승인 가능했던 문제 차단` |
| ____ | 리팩터링 | `refactor: PurchaseOrderService 상태 전이를 도메인 메서드로 이동` |
| ____ | 문서 변경 | `docs: PRD/TRD 발주 입고 상태 추가` |
| ____ | 테스트 코드 | `test: 발주 승인 상태 검증 통합 테스트 추가` |
| chore | 빌드/설정 | `chore: H2 콘솔 설정 활성화` |
| style | 코드 스타일 | `style: import 정리` |

### 작성 원칙

1. 제목은 50자 이내, 마침표 없음, 명령조("구현한다" 가 아닌 "구현").
2. body 에는 "왜" 를 적는다. "무엇" 은 diff 가 보여준다.
3. 한 커밋 = 한 가지 의도. (거래처 + 품목 + 발주를 한 커밋에 섞지 말 것)
4. 도메인 단위 커밋이 모이면 PR 의 흐름이 살아난다.

### 좋은 메시지 vs 나쁜 메시지

```text
❌ fix: 버그 수정
❌ update PurchaseOrder
❌ wip

✅ fix: REQUESTED 가 아닌 발주를 approve 호출 시 INVALID_STATUS 로 차단
✅ refactor: 발주 상태 전이를 도메인 메서드로 캡슐화하고 Service 의 if 분기 제거
✅ feat: 발주서 헤더-라인을 cascade ALL + orphanRemoval 로 정합성 보장
```
