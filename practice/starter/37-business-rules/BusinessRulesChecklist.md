# 비즈니스 규칙 체크리스트

TRD 3.8 의 모든 비즈니스 규칙을 "어디에서 검증하는가" 표로 정리해 봅니다. 빈칸을 채우고 나면, Service / Entity / DTO / DB 중 **검증 책임이 어디에 있어야 하는지** 머리에 그림이 잡힙니다.

검증 위치 표기:
- **D** = DTO (Bean Validation)
- **E** = Entity (도메인 메서드)
- **S** = Service (비즈니스 흐름)
- **DB** = DB 제약 (unique, not null)

## 7. 다층 방어 원칙

같은 규칙을 여러 계층에서 검증하는 이유:

1. **D (DTO)** — 잘못된 요청을 빨리 차단, 응답 메시지를 표준화.
2. **S (Service)** — 비즈니스 흐름의 조건(상태, 권한, 관계) 확인.
3. **E (Entity)** — 도메인 불변식. 다른 호출 경로에서도 깨지지 않도록 마지막 방어선.
4. **DB** — race condition 까지 막는 최종 무결성.
